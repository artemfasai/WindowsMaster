# Дорожня карта Windows Master

Дорожня карта описує послідовність розвитку спроможностей без фіксованих дат.
Терміни встановлюються лише після призначення власників, оцінки залежностей і
затвердження ресурсів.

## Принципи пріоритизації

Порядок пріоритетів: безпека, коректність, відновлюваність, підтримуваність,
відтворюваність і продуктивність. Кожен етап має завершуватися перевірюваними
артефактами, а не декларацією про готовність.

## Етап 1 — Фундамент workspace

- [x] Визначити кореневі правила безпеки та схвалення.
- [x] Додати інструкції для Codex і Claude Code.
- [x] Створити ролі спеціалізованих AI-агентів.
- [x] Започаткувати журнал змін і консервативну ліцензійну політику.
- [ ] Призначити власника, мейнтейнерів і канали підтримки.
- [ ] Затвердити матрицю Windows, PowerShell і Python.
- [ ] Додати шаблони ADR, runbook і нового проєкту.
- [ ] Упровадити CI для форматування, lint, тестів і secret scanning.

Критерій виходу: відтворюване середовище розробки, визначені власники та
обов'язкові quality gates.

## Етап 2 — Ядро автоматизації

- [ ] Створити нормалізований модуль discovery та inventory.
- [ ] Визначити схеми конфігурації та валідації.
- [ ] Реалізувати безпечні примітиви виконання з `-WhatIf`, timeout,
  cancellation, bounded retry, rollback і verification.
- [ ] Додати структуроване логування, correlation ID і redaction.
- [ ] Покрити ядро Pester/pytest та ізольованими інтеграційними тестами.

Критерій виходу: ідемпотентні й спостережувані операції з перевіреним відкатом.

## Етап 3 — Безпека та керування

- [ ] Інтегрувати схвалені secret stores та керовані ідентичності.
- [ ] Додати policy evaluation, RBAC і separation of duties.
- [ ] Упровадити підпис артефактів, SBOM, provenance та контроль promotion.
- [ ] Формалізувати життєвий цикл винятків і compliance evidence.

Критерій виходу: кожна привілейована дія авторизована, атрибутована й
підтверджена аудитними доказами.

## Етап 4 — Enterprise-інтеграції

- [ ] Додати версіоновані адаптери для endpoint management, directory services,
  ticketing, monitoring і backup.
- [ ] Реалізувати обмежену оркестрацію з maintenance windows, health gates і
  failure-domain controls.
- [ ] Підключити AI-планування в межах тих самих політик і схвалень, що діють
  для операторів.

Критерій виходу: кожна інтеграція має власника, опис data flow, автентифікацію,
timeout, rollback і процедуру підтримки.

## Етап 5 — Операційна зрілість

- [ ] Визначити SLI, SLO, error budgets, RTO та RPO.
- [ ] Регулярно тестувати disaster recovery і degraded operations.
- [ ] Запровадити release, deprecation, migration і periodic control review.
- [ ] Вимірювати change failure rate та ефективність автоматизації.

Критерій виходу: вимірювана надійність, протестоване відновлення та стабільне
операційне володіння.

## Керування змінами дорожньої карти

Зміни пріоритетів мають містити власника рішення, обґрунтування, залежності,
ризики та оновлені критерії виходу. Наскрізні архітектурні рішення фіксуються
ADR у `Documents/`.
